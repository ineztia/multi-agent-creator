# 引擎架构参考模板

> 本模板定义引擎实例的结构规范、核心文件格式和执行纪律。
> 生成器在构造引擎时参照本模板，并根据具体事项进行适配。

---

## 1. 核心设计原则

**动态适配原则（统领全文）**：引擎的一切设计决策均由事项属性驱动，不预设任何具体内容——维度的数量、类型、层次由事项属性决定；专家的数量、类型、配置由事项属性决定；子专题的数量和内容由事项属性决定；扩展研究是否需要及其类型由事项属性决定。

---

## 2. 目录结构

引擎实例目录为其根目录（即 `./`），内部所有路径均为相对路径。

维度编号从 `01` 开始递增，扩展研究（如有）紧随其后，综合判断、质量控制、最终报告依次递增。设维度数为 D，扩展研究占 E 个编号（无扩展研究时 E=0），则：

| 区域 | 编号范围 |
|------|---------|
| 维度研究区 | `01` ~ `{D}` |
| 扩展研究区（可选） | `{D+1}` ~ `{D+E}` |
| 综合判断区 | `{D+E+1}` |
| 质量控制区 | `{D+E+2}` |
| 最终报告区 | `{D+E+3}` |

### 2.1 完整目录结构示例

```
./
├── main.md                              # 引擎主入口
├── progress.md                          # 执行进度追踪
├── agents/                              # 专家智能体定义
│   ├── agent-01-{专家类型}.md
│   ├── agent-02-{专家类型}.md
│   └── ...
├── rules/                               # 任务专属规范
│   └── ...
├── 01-{维度1名称}/                      # 维度 1 研究区
│   ├── 01-01-{子专题1}.md
│   ├── 01-02-{子专题2}.md
│   ├── 01-{维度1名称}-综合评估.md
│   └── data/
│       └── sources-registry.md
├── 02-{维度2名称}/                      # 维度 2 研究区
│   └── ...
├── {D+1}-{扩展研究类型}/                # 扩展研究区（可选）
│   ├── {D+1}-01-{研究主题1}.md
│   └── data/
│       └── sources-registry.md
├── {D+E+1}-综合判断/                    # 综合判断区
│   ├── {D+E+1}-01-{综合判断内容1}.md
│   └── ...
├── {D+E+2}-质量控制/                    # 质量控制区
│   ├── review-round1.md
│   ├── review-round2.md
│   ├── review-round3-综合报告.md
│   └── fact-check-log.md
└── {D+E+3}-最终报告/                    # 最终报告区
    ├── {D+E+3}-{事项名称}-v1.0.md
    ├── {D+E+3}-摘要.md
    ├── {D+E+3}-附录-数据源清单与质量评级.md
    └── {D+E+3}-附录-关键学术参考文献.md
```

### 2.2 命名规范

- 目录和文件名统一使用**连字符**（`-`）作为分隔符，不使用下划线
- 编号格式：目录编号两位数字 `XX`，子文件编号 `XX-YY`
- 综合评估文件命名：`XX-{维度名称}-综合评估.md`（不使用子编号）

## 3. 核心文件规范

### 3.1 main.md（引擎主入口）

引擎的运行入口和主路由，必须包含以下六大模块：

#### 模块一：事项描述

```markdown
# {事项名称} - 多元专家研究引擎

## 事项描述

### 原始事项
[用户输入的原始事项描述]

### 解构后的研究问题
1. [研究问题1]
2. [研究问题2]
...
```

#### 模块二：专家智能体清单

```markdown
## 专家智能体清单

| 编号 | 专家类型 | 负责维度/职能 | 定义文件 |
|------|---------|-------------|---------|
| agent-01 | {类型} | {维度/职能} | agents/agent-01-{类型}.md |
| agent-02 | {类型} | {维度/职能} | agents/agent-02-{类型}.md |
| ... | ... | ... | ... |
```

#### 模块三：研究维度总览

```markdown
## 研究维度总览

### 维度间逻辑关系
[阐述维度之间的逻辑递进关系，说明维度设计如何受哲学审视结论驱动]

### 维度详情

| 维度编号 | 维度名称 | 子专题 | 负责专家 |
|---------|---------|--------|---------|
| 01 | {名称} | 01-01-{子专题1}、01-02-{子专题2}... | agent-01 |
| 02 | {名称} | 02-01-{子专题1}、02-02-{子专题2}... | agent-02 |
| ... | ... | ... | ... |
```

#### 模块四：工作流路由表

```markdown
## 工作流路由表

| 阶段 | 编号范围 | 执行者 | 产出路径 | 衔接条件 |
|------|---------|--------|---------|---------|
| 阶段一：专题研究 | 01~{D} | 各维度专家 | XX-YY-{子专题}.md、XX-{维度}-综合评估.md | 所有维度研究完成 → 判断是否需要扩展研究 |
| 阶段二：扩展研究 | {D+1}~{D+E} | 横向研究专家（如有） | {D+E}-01-{主题}.md | 扩展研究完成 → 综合判断 |
| 阶段三：综合判断 | {D+E+1} | 综合判断专家 | {D+E+1}-01-{内容}.md | 综合判断完成 → 质量审查 |
| 阶段四：质量审查 | {D+E+2} | 质量审查专家 | review-round1.md 等 | 审查通过（或达到最大审查轮次） → 最终成稿 |
| 阶段五：最终成稿 | {D+E+3} | 报告撰写专家 | {D+E+3}-{事项}-v1.0.md 等 | - |
```

#### 模块五：全局配置

```markdown
## 全局配置

- 成果形式：[学术报告 / 其他指定形式]
- 质量标准等级：[标准等级]
- 扩展研究决策：[需要/不需要，附理由]
- 规范引用：./rules/
```

#### 模块六：执行纪律与进度追踪

```markdown
## 执行纪律与进度追踪

### 执行纪律
- **逐文件串行执行**：一次只处理一个文件，每次以一位专家的身份完成一份完整文件
- **禁止批量生成**：不得在脑中「一次性构思所有维度的研究」然后批量输出
- **禁止占位内容**：最终文件不得保留任何 `[待研究]`、`[TODO]`、`[此处补充]` 等占位标记
- **禁止无推理断言**：任何结论性判断必须展示推理过程，不得直接从数据跳到结论
- **禁止信息陈列替代论证**：每个实质性章节的核心内容必须以论述文本而非纯表格/列表呈现
- **禁止单一解释**：对于重要的因果判断，必须考虑替代解释并说明取舍理由
- **禁止虚假精确**：推测性数据不得以确定性格式呈现
- **禁止结论搬运**：最终报告必须重建论证链，而非搬运子专题结论
- **禁止泛泛通用描述**：每份产出必须具体到该事项的特定语境

### 内容实质性标准
参见 ./rules/ 中的内容实质性检查项

### 阶段门控验证
参见 ./rules/ 中的阶段门控规范

### 进度追踪
参见 progress.md
```

### 3.2 agents/agent-XX-{专家类型}.md（专家智能体定义）

每份专家定义文件必须包含以下要素：

- **专家身份**：专业领域、资历背景、核心能力
- **思维立场**：该专家看问题的独特认知视角和追问方式——会问什么别人不会问的问题、会从什么角度质疑常识、面对数据不足时如何处理
- **研究范围**：负责的维度和子专题
- **研究方法**：该领域的主要研究方法和数据来源策略
- **产出要求**：应产出的文件清单和格式规范
- **质量标准**：该领域特有的质量评判标准

### 3.3 progress.md（执行进度追踪）

引擎运行期间自动维护的状态文件，用于记录工作流执行进度，支持任务中断后恢复。

**文件格式**：

```markdown
# 执行进度

## 状态概要

- 当前阶段：[阶段一：专题研究 / 阶段二：扩展研究 / ... / 阶段五：最终成稿]
- 当前专家：[agent-XX-{类型}]
- 当前文件：[正在处理的文件路径]
- 最后更新：[时间戳]

## 文件完成清单

| 阶段 | 文件 | 状态 | 完成时间 |
|------|------|------|----------|
| 阶段一 | 01-01-{子专题}.md | ✅ 已完成 / ⏳ 进行中 / ⬜ 未开始 | [时间] |
| ... | ... | ... | ... |

## 阶段门控记录

| 门控点 | 状态 | 检查时间 | 备注 |
|-------|------|---------|------|
| 阶段一 → 阶段二 | ⬜ 未到达 | - | - |
| ... | ... | ... | ... |
```

**更新规则**：

| 触发时机 | 更新内容 |
|---------|----------|
| 开始处理一份新文件前 | 将该文件状态改为 `⏳ 进行中`，更新「当前专家」和「当前文件」 |
| 一份文件写入完成后 | 将该文件状态改为 `✅ 已完成`，记录完成时间 |
| 通过阶段门控验证后 | 更新「当前阶段」，记录门控通过时间 |
| 质量审查打回修改时 | 在门控记录中标注打回信息和责任文件 |

### 3.4 data/sources-registry.md（数据源注册表）

每个维度研究区下的数据源注册表格式：

```markdown
# 数据源注册表

| 来源名称 | 类型 | 权威性评级 | 访问时间 | 对应文件 |
|---------|------|----------|---------|---------|
| ... | ... | ... | ... | ... |
```

字段说明：
- **来源名称**：数据源的完整名称
- **类型**：学术论文/行业报告/政府数据/新闻报道等
- **权威性评级**：高/中/低
- **访问时间**：数据获取时间
- **对应文件**：在 `data/` 目录中的文件名

## 4. 研究工作流

### 4.1 阶段总览

```
专题研究 ──→ [扩展研究（可选）] ──→ 综合判断 ──→ 质量审查 ──→ 最终成稿
             ↑ 由事项属性决定是否需要
```

### 4.2 专题研究阶段（必选）

**目标**：从多个维度对事项进行深度并行研究

**要求**：
- 各维度专家独立、并行研究，互不干扰
- 每个维度下设置多个子专题
- 每份独立研究需查询尽可能多的权威信息，形成研究依据

**产出**：各维度的子专题研究报告 + 维度综合评估

### 4.3 扩展研究阶段（可选）

**触发条件**：由事项属性决定

**研究类型**（非穷举）：横向对比、案例深潜、趋势推演、跨域关联等

**要求**：
- 允许跨维度综合信息
- 应发现专题研究阶段未覆盖的洞察

**产出**：扩展研究报告

### 4.4 综合判断阶段（必选）

**目标**：整合所有研究成果，形成整体判断

**要求**：
- 整合专题研究和扩展研究（如有）的成果
- 回答核心问题，形成整体判断
- 研判未来趋势，提出建议

**产出**：综合判断报告

### 4.5 质量审查阶段（必选）

详见 `quality-assurance.md` 中的三级审查规范。

### 4.6 最终成稿阶段（必选）

**目标**：构建整个研究的顶层论证——读者仅阅读最终报告，即可理解从证据到核心结论的完整推理过程

**核心要求——论证重建而非结论搬运**：

最终报告不是将子专题的结论搬运到一个文件中，而是将分散在子专题中的推理链**重建**为一个连贯的顶层论证。

**具体要求**：
1. 关键推理链必须在最终报告中完整呈现——每个核心策略建议必须展示从制度事实→机制推理→约束条件→修正结论的完整路径
2. 约束条件和边界条件不可省略——子专题中识别出的约束必须在最终报告中保留
3. 争议和不确定性不可省略——子专题中讨论的替代解释、估算的不确定性、数据的局限，必须在最终报告中保留
4. 摘要是最终报告的压缩版（不是结论清单），附录提供数据细节

**产出**：最终研究报告、摘要、附录

### 4.7 阶段衔接规则

| 前序阶段 | 衔接条件 | 后续阶段 |
|---------|---------|---------|
| 专题研究 | 所有维度研究完成 → 判断是否需要扩展研究 | 扩展研究 或 综合判断 |
| 扩展研究 | 扩展研究完成 | 综合判断 |
| 综合判断 | 综合判断完成 | 质量审查 |
| 质量审查 | 审查通过（或达到最大审查轮次） | 最终成稿 |

## 5. 执行纪律

### 5.1 逐文件串行执行

- **一次只处理一个文件**：每次以一位专家的身份，完成一份子专题研究报告的完整写作，写入文件后，才能开始下一份
- **禁止批量生成**：不得在脑中「一次性构思所有维度的研究」然后批量输出
- **禁止占位内容**：最终文件不得保留任何 `[待研究]`、`[TODO]`、`[此处补充]` 等占位标记

### 5.2 阶段门控验证

进入下一阶段前，执行 AI 必须逐一检查前序阶段的每份产出文件：

| 门控点 | 验证内容 | 不通过处理 |
|-------|---------|-----------|
| 专题研究 → 下一阶段 | 每份子专题报告和综合评估是否满足内容实质性标准 | 回到未完成的文件继续研究 |
| 扩展研究 → 综合判断 | 扩展研究报告是否包含对专题研究成果的实际引用 | 回到扩展研究补充 |
| 综合判断 → 质量审查 | 综合判断报告是否实际整合了前序阶段的具体发现 | 回到综合判断补充 |

### 5.3 强制执行顺序（含进度追踪）

```
0. 启动时检查 progress.md → 若存在则执行恢复协议 → 从中断点继续

对每个维度 D_i（i = 1..D）：
  1. 读取该维度专家定义文件，进入专家角色
  对 D_i 下的每个子专题 S_j（j = 1..N）：
    2. 更新 progress.md：当前文件 → ⏳ 进行中
    3. 针对子专题 S_j 进行独立研究
    4. 将完整研究报告写入 XX-YY-{子专题}.md
    5. 更新 data/sources-registry.md
    6. 更新 progress.md：当前文件 → ✅ 已完成
  完成所有子专题后：
    7. 更新 progress.md：综合评估文件 → ⏳ 进行中
    8. 基于已完成的子专题报告，撰写维度综合评估
    9. 将综合评估写入 XX-{维度名称}-综合评估.md
   10. 更新 progress.md：综合评估文件 → ✅ 已完成

所有维度完成 → 门控验证 → 更新 progress.md → 进入下一阶段
```

### 5.4 恢复协议

执行 AI 在启动时（包括中断后重新启动），必须执行以下恢复流程：

1. 检查 `progress.md` 是否存在：若存在，说明是中断恢复场景
2. 读取状态概要：确认当前阶段和当前文件
3. 扫描文件完成清单：找到第一个状态为 `⏳ 进行中` 或 `⬜ 未开始` 的文件
4. 从该文件继续执行：如果文件状态为 `⏳ 进行中`，检查文件内容——若已有实质内容则标记完成并继续下一个，若仍为骨架则重新研究该文件
5. 不得重复已完成的工作：状态为 `✅ 已完成` 的文件直接跳过

### 5.5 研究迭代机制

当前「逐文件串行、写完即完成」的执行纪律确保了研究过程的有序性，但不应成为思维深度的桎梏。以下场景允许且鼓励回溯修正：

- **维度综合评估阶段**：撰写综合评估时，若发现子专题报告中存在不准确、论证链断裂或结论需要修正之处，应回溯修正子专题报告
- **综合判断阶段**：若发现前序研究存在明显薄弱环节（如某个关键因果链缺乏论证），应触发补充研究
- **质量审查阶段**：审查发现「论证链断裂」「估算值无依据」时，应打回要求补充推理，而非仅打回格式

回溯修正时，在 `progress.md` 中记录修正原因和涉及的文件。

### 5.6 严禁行为清单

- 创建完目录结构后直接跳到综合判断或最终报告
- 用概括性语言代替实际研究
- 多个维度共享相同的分析框架和结论
- 在未完成前序阶段全部产出的情况下启动后续阶段
- 禁止无推理断言：任何结论性判断必须展示推理过程，不得直接从数据跳到结论
- 禁止信息陈列替代论证：每个实质性章节的核心内容必须以论述文本而非纯表格/列表呈现
- 禁止单一解释：对于重要的因果判断，必须考虑替代解释并说明取舍理由
- 禁止虚假精确：推测性数据不得以确定性格式呈现

## 6. 产出层次结构

引擎采用三层递进的产出结构：

| 层级 | 产出类型 | 说明 | 文件命名 |
|------|---------|------|----------|
| 第一层 | 子专题研究报告 | 可独立阅读，包含完整研究过程和结论 | `XX-YY-{专题名称}.md` |
| 第二层 | 维度综合评估 | 整合该维度下所有子专题的研究成果 | `XX-{维度名称}-综合评估.md` |
