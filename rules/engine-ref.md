# 引擎规范参考模板

> **模板类型**：L2 转义模板（从 L1 引擎规范转义而来）
> **用途**：供生成器构造引擎时参考，并作为动态生成 `{TASK}/rules/engine-ref.md` 的基础
> **使用方式**：生成器将本模板中的变量占位符替换为具体事项信息，生成任务专属版本

---

## 1. 目录结构模板

引擎实例目录位于 `tasks/{UNIX时间戳}-{TASK}/`。

设维度数为 `D`，扩展研究占 `E` 个编号（无扩展研究时 `E=0`）：

| 区域 | 编号范围 | 说明 |
|------|---------|------|
| 维度研究区 | `01` ~ `{D}` | 每个维度独立研究 |
| 扩展研究区 | `{D+1}` ~ `{D+E}` | 可选，跨维度综合 |
| 综合判断区 | `{D+E+1}` | 整合所有研究成果 |
| 质量控制区 | `{D+E+2}` | 三级审查 + 事实核查 |
| 最终报告区 | `{D+E+3}` | 完整交付物 |

```
tasks/{TASK}-{TS}/
├── main.md
├── progress.md
├── agents/
│   └── agent-{NN}-{专家类型}.md
├── rules/
│   ├── engine-ref.md
│   ├── research-ref.md
│   ├── data-source-eval.md
│   └── quality-assurance.md
├── 01-{维度名称}/
│   ├── 01-01-{子专题}.md
│   ├── 01-{维度名称}-综合评估.md
│   └── data/sources-registry.md
├── ...
├── {D+E+1}-综合判断/
├── {D+E+2}-质量控制/
│   ├── review-round1.md
│   ├── review-round2.md
│   ├── review-round3-综合报告.md
│   └── fact-check-log.md
└── {D+E+3}-最终报告/
    ├── {D+E+3}-{事项名称}-v1.0.md
    ├── {D+E+3}-摘要.md
    ├── {D+E+3}-附录-数据源清单与质量评级.md
    └── {D+E+3}-附录-关键学术参考文献.md
```

**命名规范**：
- 连字符（`-`）作为分隔符，不使用下划线
- 目录编号两位数字 `XX`，子文件编号 `XX-YY`
- 综合评估：`XX-{维度名称}-综合评估.md`


## 3. 工作流阶段与门控

### 3.1 五阶段工作流

```
专题研究 ──→ [扩展研究] ──→ 综合判断 ──→ 质量审查 ──→ 最终成稿
```

### 3.2 阶段衔接条件

| 前序阶段 | 衔接条件 | 后续阶段 |
|---------|---------|---------|
| 专题研究 | 所有维度研究完成 → 判断是否需要扩展研究 | 扩展研究 或 综合判断 |
| 扩展研究 | 扩展研究完成 | 综合判断 |
| 综合判断 | 综合判断完成 | 质量审查 |
| 质量审查 | 审查通过（或达到最大审查轮次） | 最终成稿 |

### 3.3 门控验证方法

| 门控点 | 验证内容 | 不通过处理 |
|-------|---------|-----------|
| 专题 → 下一阶段 | 每份产出满足内容实质性标准 | 回到未完成文件继续 |
| 扩展 → 综合判断 | 扩展报告包含对专题成果的实际引用 | 回到扩展研究补充 |
| 综合 → 质量审查 | 综合报告实际整合了前序阶段具体发现 | 回到综合判断补充 |

---

## 4. 执行纪律要求

### 4.1 逐文件串行执行

- 一次只处理一个文件
- 禁止批量生成
- 禁止占位内容（不得保留 `[待研究]`、`[TODO]` 等标记）

### 4.2 内容实质性标准

| 产出类型 | 最低篇幅 | 必备章节 | 数据要求 |
|---------|---------|---------|---------|
| 子专题研究报告 | 800 字 | 研究问题、方法、发现、论证、结论 | ≥3 个数据来源 |
| 维度综合评估 | 1000 字 | 子专题发现、交叉分析、维度判断、洞察 | 逐一引用每份子专题 |
| 扩展研究报告 | 800 字 | 目标、跨维度分析、发现、结论 | 引用专题研究成果 |
| 综合判断报告 | 1500 字 | 成果整合、核心回答、趋势、建议 | 引用前序数据和结论 |

### 4.3 强制执行顺序

```
0. 启动时检查 progress.md → 若存在则执行恢复协议 → 从中断点继续

对每个维度 D_i：
  1. 读取该维度专家定义，进入专家角色
  对 D_i 下每个子专题 S_j：
    2. 更新 progress.md → ⏳ 进行中
    3. 独立研究子专题
    4. 写入完整报告
    5. 更新 sources-registry.md
    6. 更新 progress.md → ✅ 已完成
  完成所有子专题：
    7-10. 撰写维度综合评估

所有维度完成 → 门控验证 → 进入下一阶段
```

### 4.4 严禁行为

- 创建完目录后直接跳到综合判断或最终报告
- 用概括性语言代替实际研究
- 多个维度共享相同的分析框架和结论
- 在未完成前序阶段全部产出的情况下启动后续阶段

### 4.5 恢复协议

1. 检查 `progress.md` 是否存在
2. 读取状态概要，确认当前阶段和当前文件
3. 找到第一个 `⏳ 进行中` 或 `⬜ 未开始` 的文件
4. 从该文件继续执行
5. 不重复已完成的工作
