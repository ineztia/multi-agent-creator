# 引擎架构与执行纪律参考

> 本文件是生成器构造引擎实例时的架构参考模板。生成器在构造 `{TASK}/rules/` 时，基于本模板适配具体事项。

---

## 1. 引擎目录结构

引擎实例的工作目录为其根目录（`./`），内部所有路径均为相对路径。目录的创建位置和命名由生成器决定，引擎无需感知。

维度编号从 `01` 开始递增，扩展研究（如有）紧随其后，综合判断、质量控制、最终报告依次递增。设维度数为 D，扩展研究占 E 个编号（无扩展研究时 E=0）：

| 区域 | 编号范围 |
|------|---------|
| 维度研究区 | `01` ~ `{D}` |
| 扩展研究区（可选） | `{D+1}` ~ `{D+E}` |
| 综合判断区 | `{D+E+1}` |
| 质量控制区 | `{D+E+2}` |
| 最终报告区 | `{D+E+3}` |

```
./
├── main.md                              # 引擎主入口
├── progress.md                          # 执行进度追踪
├── agents/                              # 专家智能体定义
│   ├── agent-01-{专家类型}.md
│   ├── agent-02-{专家类型}.md
│   └── ...
├── rules/                               # 任务专属规范（文件组织由生成器自主设计）
│   └── ...
├── 01-{维度1名称}/                      # 维度 1 研究区
│   ├── 01-01-{子专题1}.md
│   ├── 01-02-{子专题2}.md
│   ├── 01-{维度1名称}-综合评估.md
│   └── data/
│       └── sources-registry.md
├── 02-{维度2名称}/                      # 维度 2 研究区
│   └── ...
├── {D+1}-{扩展研究类型}/                # 扩展研究区（可选）
│   ├── {D+1}-01-{研究主题1}.md
│   └── data/
│       └── sources-registry.md
├── {D+E+1}-综合判断/                    # 综合判断区
│   ├── {D+E+1}-01-{综合判断内容1}.md
│   └── ...
├── {D+E+2}-质量控制/                    # 质量控制区
│   ├── review-round1.md
│   ├── review-round2.md
│   ├── review-round3-综合报告.md
│   └── fact-check-log.md
└── {D+E+3}-最终报告/                    # 最终报告区
    ├── {D+E+3}-{事项名称}-v1.0.md
    ├── {D+E+3}-摘要.md
    ├── {D+E+3}-附录-数据源清单与质量评级.md
    └── {D+E+3}-附录-关键学术参考文献.md
```

### 命名规范

- 目录和文件名统一使用**连字符**（`-`）作为分隔符，不使用下划线
- 编号格式：目录编号两位数字 `XX`，子文件编号 `XX-YY`
- 综合评估文件命名：`XX-{维度名称}-综合评估.md`（不使用子编号）
- 此规范适用于引擎内所有层级

## 2. 核心文件规范

### 2.1 main.md（引擎主入口）

引擎的运行入口和主路由，必须包含以下六大模块：

| 模块 | 内容要求 |
|------|---------|
| 事项描述 | 用户输入的原始事项和解构后的研究问题 |
| 专家智能体清单 | 列出所有专家及其负责的维度，引用 `agents/` 下的定义文件 |
| 研究维度总览 | 各维度名称、子专题列表、对应专家、维度间逻辑关系 |
| 工作流路由表 | 各阶段的执行顺序、衔接条件、文件产出路径 |
| 全局配置 | 成果形式、质量标准等级、扩展研究决策结果、规范引用指向 `./rules/` |
| 执行纪律与进度追踪 | 引用 `./rules/` 中的引擎执行纪律参考，说明逐文件串行执行规则、内容实质性标准、阶段门控验证方法和恢复协议 |

### 2.2 agents/agent-XX-{专家类型}.md（专家智能体定义）

每份专家定义文件必须包含：

| 字段 | 说明 |
|------|------|
| 专家身份 | 专业领域、资历背景、核心能力 |
| 研究范围 | 负责的维度和子专题 |
| 研究方法 | 该领域的主要研究方法和数据来源策略 |
| 产出要求 | 应产出的文件清单和格式规范 |
| 质量标准 | 该领域特有的质量评判标准 |

**禁止泛泛通用描述**：每位专家的身份、方法、数据源策略必须与其负责的维度深度匹配，不同专家之间必须有明显的能力差异化。

### 2.3 progress.md（执行进度追踪）

引擎运行期间自动维护的状态文件，用于记录工作流执行进度，支持任务中断后恢复。

**文件格式**：

```markdown
# 执行进度

## 状态概要

- 当前阶段：[阶段一：专题研究 / 阶段二：扩展研究 / ... / 阶段五：最终成稿]
- 当前专家：[agent-XX-{类型}]
- 当前文件：[正在处理的文件路径]
- 最后更新：[时间戳]

## 文件完成清单

| 阶段 | 文件 | 状态 | 完成时间 |
|------|------|------|----------|
| 阶段一 | 01-01-{子专题}.md | ✅ 已完成 / ⏳ 进行中 / ⬜ 未开始 | [时间] |
| ... | ... | ... | ... |

## 阶段门控记录

| 门控点 | 状态 | 检查时间 | 备注 |
|-------|------|---------|------|
| 阶段一 → 阶段二 | ⬜ 未到达 | - | - |
| ... | ... | ... | ... |
```

**更新规则**：

| 触发时机 | 更新内容 |
|---------|----------|
| 开始处理一份新文件前 | 将该文件状态改为 `⏳ 进行中`，更新「当前专家」和「当前文件」 |
| 一份文件写入完成后 | 将该文件状态改为 `✅ 已完成`，记录完成时间 |
| 通过阶段门控验证后 | 更新「当前阶段」，记录门控通过时间 |
| 质量审查打回修改时 | 在门控记录中标注打回信息和责任文件 |

**恢复协议**：

执行 AI 在启动时（包括中断后重新启动），必须执行以下恢复流程：

1. **检查 `progress.md` 是否存在**：若存在，说明是中断恢复场景
2. **读取状态概要**：确认当前阶段和当前文件
3. **扫描文件完成清单**：找到第一个状态为 `⏳ 进行中` 或 `⬜ 未开始` 的文件
4. **从该文件继续执行**：如果文件状态为 `⏳ 进行中`，检查文件内容——若已有实质内容则标记完成并继续下一个，若仍为骨架则重新研究该文件
5. **不得重复已完成的工作**：状态为 `✅ 已完成` 的文件直接跳过

### 2.4 data/sources-registry.md（数据源注册表）

每个维度研究区下的数据源注册表，格式：

| 字段 | 说明 |
|------|------|
| 来源名称 | 数据源的完整名称 |
| 类型 | 学术论文/行业报告/政府数据/新闻报道等 |
| 权威性评级 | 高/中/低（评级标准见 `./rules/` 中的数据源评估参考） |
| 访问时间 | 数据获取时间 |
| 对应文件 | 在 `data/` 目录中的文件名 |

## 3. 研究维度与专家体系

### 3.1 维度设计要求

- 每个维度配置专门的专家智能体
- 每个维度下设置多个子专题，确保研究深度
- 维度的数量、类型、层次完全由事项属性驱动

### 3.2 维度研究独立性

- **专题研究阶段**：各维度专家独立研究，互不干扰。每位专家独立采集数据、独立论证、独立产出研究报告，均指向核心问题，不做维度间横向交叉
- **扩展研究阶段**（例外）：扩展研究允许且鼓励跨维度综合

---

## 4. 产出层次结构

引擎采用递进的产出结构：

| 层级 | 产出类型 | 说明 | 文件命名 |
|------|---------|------|----------|
| 第一层 | 子专题研究报告 | 可独立阅读，包含完整研究过程和结论 | `XX-YY-{专题名称}.md` |
| 第二层 | 维度综合评估 | 整合该维度下所有子专题的研究成果 | `XX-{维度名称}-综合评估.md` |
| 第二层 | 扩展研究报告 | 跨维度综合分析（如有） | `{D+1}-01-{研究主题}.md` 等 |
| 第三层 | 综合判断报告 | 整合所有研究成果的整体判断 | `{D+E+1}-01-{内容}.md` 等 |
| 第三层 | 最终报告 | 完整的研究成果交付物，含报告、摘要、附录 | `{D+E+3}-{事项名称}-v1.0.md` 等 |

## 5. 五阶段研究工作流

```
专题研究 ──→ [扩展研究（可选）] ──→ 综合判断 ──→ 质量审查 ──→ 最终成稿
             ↑ 由事项属性决定是否需要
```

### 5.1 专题研究阶段（必选）

- **目标**：从多个维度对事项进行深度并行研究
- **要求**：各维度专家独立、并行研究，互不干扰；每个维度下设置多个子专题；每份独立研究需查询尽可能多的权威信息
- **产出**：各维度的子专题研究报告 + 维度综合评估

### 5.2 扩展研究阶段（可选）

- **触发条件**：由事项属性决定
- **研究类型**（非穷举）：横向对比、案例深潜、趋势推演、跨域关联等
- **要求**：允许跨维度综合信息；应发现专题研究阶段未覆盖的洞察
- **产出**：扩展研究报告

### 5.3 综合判断阶段（必选）

- **目标**：整合所有研究成果，形成整体判断
- **要求**：整合专题研究和扩展研究（如有）的成果；回答核心问题；研判未来趋势，提出建议
- **产出**：综合判断报告

### 5.4 质量审查阶段（必选）

- **规范**：详见 `./rules/` 中的质量保证参考
- **产出**：三轮审查报告 + 事实核查日志

### 5.5 最终成稿阶段（必选）

- **目标**：形成多层次的研究成果交付物
- **要求**：整合所有研究成果，形成完整报告；编写独立摘要（2-3 页）；整理附录材料
- **产出**：最终研究报告、摘要、附录

### 5.6 阶段衔接规则

| 前序阶段 | 衔接条件 | 后续阶段 |
|---------|---------|---------|
| 专题研究 | 所有维度研究完成 → 判断是否需要扩展研究 | 扩展研究 或 综合判断 |
| 扩展研究 | 扩展研究完成 | 综合判断 |
| 综合判断 | 综合判断完成 | 质量审查 |
| 质量审查 | 审查通过（或达到最大审查轮次） | 最终成稿 |

## 6. 执行纪律（防跳步规范）

> **为什么需要本节**：部分 AI 在执行引擎时，会创建目录和文件骨架后，跳过实际研究过程，直接生成最终报告。以下执行纪律确保每个研究步骤被真正执行。

### 6.1 逐文件串行执行

- **一次只处理一个文件**：每次以一位专家的身份，完成一份子专题研究报告的完整写作，写入文件后，才能开始下一份
- **禁止批量生成**：不得在脑中「一次性构思所有维度的研究」然后批量输出
- **禁止占位内容**：最终文件不得保留任何 `[待研究]`、`[TODO]`、`[此处补充]` 等占位标记

### 6.2 内容实质性标准

每份产出文件必须满足以下最低要求，否则视为未完成：

| 产出类型 | 最低篇幅 | 必备章节 | 数据要求 |
|---------|---------|---------|---------|
| 子专题研究报告 | 2000 字正文 | 研究问题、研究方法、核心发现、分析论证、结论 | 至少引用 3 个明确的数据来源或文献 |
| 维度综合评估 | 2000 字正文 | 各子专题发现摘要、交叉分析、维度整体判断、关键洞察 | 必须逐一引用该维度下每份子专题的具体发现 |
| 扩展研究报告 | 2000 字正文 | 研究目标、跨维度分析、发现与洞察、结论 | 必须实际引用专题研究阶段的具体成果 |
| 综合判断报告 | 2000 字正文 | 成果整合、核心问题回答、趋势研判、建议 | 必须引用前序阶段的具体数据和结论 |

**独立可读原则**：任何一份报告抽出来单独阅读，应能理解其研究问题和结论。

### 6.3 阶段门控验证

进入下一阶段前，执行 AI 必须逐一检查前序阶段的每份产出文件：

| 门控点 | 验证内容 | 不通过处理 |
|-------|---------|-----------|
| 专题研究 → 下一阶段 | 每份子专题报告和综合评估是否满足 §6.2 标准 | 回到未完成的文件继续研究 |
| 扩展研究 → 综合判断 | 扩展研究报告是否包含对专题研究成果的实际引用 | 回到扩展研究补充 |
| 综合判断 → 质量审查 | 综合判断报告是否实际整合了前序阶段的具体发现 | 回到综合判断补充 |

### 6.4 强制执行顺序（含进度追踪）

```
0. 启动时检查 progress.md → 若存在则执行恢复协议（§2.3）→ 从中断点继续

对每个维度 D_i（i = 1..D）：
  1. 读取该维度专家定义文件，进入专家角色
  对 D_i 下的每个子专题 S_j（j = 1..N）：
    2. 更新 progress.md：当前文件 → ⏳ 进行中
    3. 针对子专题 S_j 进行独立研究
    4. 将完整研究报告写入 XX-YY-{子专题}.md
    5. 更新 data/sources-registry.md
    6. 更新 progress.md：当前文件 → ✅ 已完成
  完成所有子专题后：
    7. 更新 progress.md：综合评估文件 → ⏳ 进行中
    8. 基于已完成的子专题报告，撰写维度综合评估
    9. 将综合评估写入 XX-{维度名称}-综合评估.md
   10. 更新 progress.md：综合评估文件 → ✅ 已完成

所有维度完成 → 门控验证（§6.3）→ 更新 progress.md → 进入下一阶段
```

**严禁以下行为**：
- 创建完目录结构后直接跳到综合判断或最终报告
- 用概括性语言代替实际研究
- 多个维度共享相同的分析框架和结论
- 在未完成前序阶段全部产出的情况下启动后续阶段

---

## 7. 层间隔离约束

每层自包含，不得引用其他层的文件。同层内文件可互相引用。

| 层级 | 引用范围 |
|------|---------|
| L3 引擎层 | 仅引用 `{TASK}/` 内文件，不得引用生成器层或元指令层 |
| L4 成果层 | 为最终研究成果，无引用需求 |

引擎执行 AI 只需读取 `{TASK}/` 内的文件即可工作，无需访问任何外部文件。
