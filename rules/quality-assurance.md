# 质量保证流程模板

> **模板类型**：L2 操作模板
> **用途**：供生成器构造引擎时参考，并作为动态生成 `{TASK}/rules/quality-assurance.md` 的基础
> **使用方式**：合并了三级审查流程和事实核查流程

---

## 1. 三级审查流程

### 1.1 审查总览

```
第一轮（结构审查）──→ 第二轮（内容审查）──→ 第三轮（综合评审）
      ↓ 不合格                ↓ 不合格                ↓
    打回修改              打回修改            标注遗留问题
```

**终止条件**：单份文件最多打回 2 次（共 3 次提交机会）。第 3 次提交后无论结果如何，标注遗留问题并继续流程。

### 1.2 第一轮：结构完整性审查

**审查重点**：产出是否齐全、格式是否合规

**检查清单**：
- [ ] 所有维度研究区均有子专题报告文件
- [ ] 所有维度研究区均有综合评估文件
- [ ] 每份报告包含完整的章节结构（摘要、研究问题、方法、发现、分析、结论）
- [ ] 每个研究区的 `data/sources-registry.md` 已填写
- [ ] 文件命名符合规范
- [ ] 扩展研究区（如有）报告已产出
- [ ] 综合判断区报告已产出

**产出文件**：`review-round1.md`

**格式**：

```markdown
# 第一轮审查报告 - 结构完整性

## 审查结果：通过 / 不通过

## 问题清单

| 编号 | 文件 | 问题描述 | 严重程度 | 责任专家 |
|------|------|---------|---------|---------|

## 审查意见
[整体评价和建议]
```

### 1.3 第二轮：内容质量审查

**审查重点**：论证是否严密、数据是否可靠、结论是否合理

**检查清单**：
- [ ] 每个关键论断有明确的数据或文献支撑
- [ ] 数据来源权威性评级合理
- [ ] 论证过程逻辑严密，无跳跃性结论
- [ ] 结论有充分的证据支撑，无过度推断
- [ ] 同一维度内各子专题之间逻辑一致
- [ ] 综合评估准确反映了各子专题的研究发现
- [ ] 对争议性问题呈现了多方观点

**产出文件**：`review-round2.md`（格式同上）

### 1.4 第三轮：综合评审

**审查重点**：整体一致性、创新性、实用价值

**检查清单**：
- [ ] 各维度研究之间逻辑一致，无相互矛盾
- [ ] 综合判断合理整合了各维度研究成果
- [ ] 核心研究问题得到了充分回答
- [ ] 研究具有独到见解和创新点
- [ ] 研究成果对目标读者有实用价值
- [ ] 整体报告结构完整、层次清晰
- [ ] 所有事实核查通过

**产出文件**：`review-round3-综合报告.md`

**格式**：

```markdown
# 第三轮审查报告 - 综合评审

## 总体评价
[整体质量评估，200-300字]

## 各维度质量评分

| 维度 | 结构完整性 | 论证质量 | 数据可靠性 | 创新性 | 综合评分 |
|------|-----------|---------|-----------|-------|---------|

## 突出优点
[列举研究的亮点]

## 遗留问题
[列举尚存的问题及处理建议]

## 审查结论
通过 / 有条件通过（附遗留问题清单）
```

## 2. 事实核查流程

### 2.1 核查范围

| 核查对象 | 优先级 | 说明 |
|---------|-------|------|
| 关键数据和统计数字 | **高** | 直接支撑核心结论的定量数据 |
| 因果关系声明 | **高** | 「A 导致了 B」类论断 |
| 引用和归因 | **中** | 「据 XX 报告...」「XX 认为...」 |
| 时间线和事件 | **中** | 历史事件的日期、顺序、参与方 |
| 专业术语和定义 | **低** | 专业概念的使用是否准确 |

### 2.2 核查方法

**交叉验证**：
- 每个关键数据至少用 2 个独立来源交叉验证
- 如果只有 1 个来源，在报告中标注「单一来源，未经交叉验证」

**来源追溯**：
- 追溯到数据的原始发布方（非二手引用）
- 确认数据发布时间和版本
- 检查数据是否已被更新或撤回

**逻辑一致性检查**：
- 同一报告内的数据是否前后一致
- 不同维度报告中引用的同一数据是否一致
- 结论是否与呈现的数据相符

### 2.3 fact-check-log.md 格式

```markdown
# 事实核查日志

## 核查概要

- 核查范围：[覆盖的文件清单]
- 核查日期：[日期]
- 核查结果：通过 / 有问题

## 核查记录

| 编号 | 文件 | 核查内容 | 原始声明 | 核查结果 | 来源验证 | 处理建议 |
|------|------|---------|---------|---------|---------|---------|

## 需修正的问题
[列出需要修正的具体问题和责任专家]

## 无法验证的声明
[列出因客观条件无法验证的声明，标注处理方式]
```

### 2.4 核查结果处理

| 结果 | 处理方式 |
|------|---------|
| ✅ 准确 | 无需处理 |
| ⚠️ 部分准确 | 要求责任专家修正不准确部分 |
| ❌ 不准确 | 要求责任专家删除或替换，提供正确数据 |
| ❓ 无法验证 | 在报告中标注「该数据未经独立验证」 |

### 2.5 核查优先级策略

当时间/资源有限时，按以下优先级执行核查：

1. **最高优先**：直接支撑最终报告核心结论的关键数据
2. **高优先**：综合判断中使用的跨维度整合数据
3. **中优先**：各维度综合评估中的重要数据
4. **低优先**：子专题报告中的辅助性数据

---

## 3. 打回修改流程

当审查不通过时：

1. 在审查报告中明确指出问题所在及对应的责任专家
2. 将审查报告发送给责任专家
3. 责任专家针对指出的问题进行修改
4. 修改后重新提交审查
5. 审查者仅复核修改部分，无需全量重审
