# 生成器规范

> 元声明
> - 版本：v1.0
> - 修订日期：2026-04-21
> - 实体层级：L1 元指令层
> - 定义对象：L2 生成器（AGENTS.md）
> - 用途：定义生成器应具备的能力、方法论和约束
> - 边界：本文件只定义「生成器应该是什么样」，不定义引擎规范（见 `L1-engine-spec.md`）或研究成果规范（见 `L1-research-spec.md`）
> - 共享规范：内容实质性标准、数据源评估原则见 `L1-common.md`

---

## 1. 生成器是什么

### 1.1 角色定义

生成器是一个提示词系统（AGENTS.md），其唯一职责是：

**接收用户输入的任意事项 → 分析事项属性 → 构造一个适配该事项的多元专家研究引擎实例 → 输出可被 AI 直接执行的引擎文件**

### 1.2 职责边界

**负责**：
- 接收并理解用户输入的事项
- 分析事项的属性、范围、复杂度
- 设计适配该事项的研究维度体系
- 为每个维度配置专门的专家智能体
- 规划完整的研究工作流
- 生成引擎实例文件（含任务专属 rules/）
- 确保引擎可被另一个 AI 直接加载执行

**不负责**：
- 不执行任何研究工作
- 不产出研究报告或研究成果
- 不与用户讨论研究内容或结论
- 不发表对研究主题的观点或判断
- 不负责验证 L1 层规范的正确性（若发现矛盾或遗漏，按 `L1-agent-builder.md` §2.5 的冲突解决协议处理）

### 1.3 核心约束（宪法级）

以下约束优先级高于本文件中的任何其他条款。当其他条款与以下约束冲突时，以此为准：

1. **不硬编码**——不预设研究类型、维度数量、专家配置、子专题内容。一切由事项属性驱动
2. **不套案例**——参考案例仅用于理解形态，每个事项的引擎设计均从零推导
3. **不越界**——只产出引擎定义，不执行研究、不产出成果、不讨论研究内容
4. **不重复**——通过 `rules/` 引用规范，不在引擎文件中重新定义已有规范
5. **不模糊**——引擎中的每条指令都应明确、具体、可执行，AI 读后无歧义

## 2. 生成什么

### 2.1 引擎实例交付物

生成器为每个事项产出一套完整的引擎实例文件，存放于 `tasks/{UNIX时间戳}-{TASK}/`（时间戳通过工具调用获取**运行时真实值**）：

| 交付物 | 说明 |
|-------|------|
| `main.md` | 引擎主入口，包含完整的任务定义和工作流路由 |
| `progress.md` | 执行进度追踪文件（初始状态，全部文件均为未开始） |
| `agents/agent-{NN}-{类型}.md` | 所有专家智能体定义文件 |
| `rules/` | 任务专属规范参考模板（从 L2 通用模板动态适配） |
| `XX-{维度名称}/` | 各维度研究区目录（含子文件骨架和 `data/sources-registry.md`） |
| `{D+E+1}-综合判断/` | 综合判断区目录 |
| `{D+E+2}-质量控制/` | 质量控制区目录 |
| `{D+E+3}-最终报告/` | 最终报告区目录 |

### 2.2 任务专属 rules/

生成器必须为每个任务动态生成 `{TASK}/rules/` 目录，使 L3 引擎层完全自包含。

任务专属 rules/ 的文件数量、命名和组织方式由生成器根据具体事项自主设计，不必镜像 L2 层 rules/ 的结构。核心要求：

- 基于 L2 `rules/` 通用模板适配，融入具体事项的领域特征（如该领域的权威数据源、常用研究方法、特有质量标准等）
- 确保 L3 层引擎完全自包含，引擎 AI 只需读取 `{TASK}/` 内的文件即可工作
- 文件组织应服务于引擎 AI 的实际工作流程，而非照搬通用模板

## 3. 引擎生成方法论

### 3.1 输入处理

#### 事项接收

用户将以自然语言描述一个研究事项。事项类型不限：
- 决策事项——「是否应该进入东南亚市场」
- 科研课题——「量子计算对现代密码学的影响」
- 对比分析——「中美 AI 竞争力与产业应用对比」
- 问题解答——「全球粮食安全面临的主要威胁」
- 趋势研判——「未来五年新能源汽车产业格局」
- 其他任意需求

> 若用户输入不属于以上任何明确类型，按**研究型**处理，默认以学术报告格式输出。

#### 充分性检查

**最低信息要求**：事项主题明确，且可推导出至少 2 个有意义的研究维度。

**追问机制**：如果输入不足，一次性向用户追问以下问题（按需选择，不必全问）：
- 研究的核心问题是什么？希望最终回答什么问题？
- 期望的研究深度和广度？
- 是否有特别关注的维度、角度或领域？
- 研究成果的预期用途？（决策支持/学术发表/内部参考/...）
- 是否有特殊的输出格式要求？（默认为学术报告格式）

追问应简洁直接，**一次性提出，最多追问 3 轮**。若用户未回应追问，按已有信息继续构造引擎。

#### 成果形式确认

默认以符合学术期刊标准的学术报告呈现。用户另有指定时从其指定。在引擎的 `main.md` 全局配置中记录该决策。

### 3.2 事项解构

#### 3.2.1 属性分析

从以下角度分析事项属性，作为后续一切设计决策的驱动输入：

| 属性 | 分析要点 |
|------|---------|
| 领域 | 事项涉及的学科或行业领域（可能跨域） |
| 类型 | 决策型 / 研究型 / 对比型 / 探索型 / 评估型 / 综合型 |
| 范围 | 地理范围、时间跨度、行业范围等 |
| 复杂度 | 涉及学科数、利益相关方数量、争议程度、数据可得性 |
| 关键张力 | 事项中存在的核心矛盾、对立面或权衡取舍 |
| 哲学审视 | 见「哲学审视方法论」章节（本节后） |

---

#### 3.2.2 哲学审视方法论

哲学审视不是装饰性文字，而是必须实质影响维度设计的分析步骤。其本质是从宏观理论视角审视事项的深层结构——追问被默认接受的前提假设、识别隐含的价值立场与认知偏见、区分不可化约的本质约束与可改变的表象条件、探查知识边界与认知盲区。

**核心原则**：哲学审视不设固定框架。针对具体事项，从宏观理论视角出发，选取最能穿透其表象的哲学视角。以下清单提供宏观理论视角的示例，但不应局限于此——应根据事项的独特属性，动态拆解更多哲学维度。

| 哲学理论视角 | 核心问题 |
|-------------|---------|
| 矛盾论 | 事项中对立面如何统一？主要矛盾与次要矛盾是什么？矛盾双方的力量对比和转化条件是什么？ |
| 认识论 | 该事项的知识是如何被建构的？经验知识与先验知识的边界在哪里？可知与不可知的边界是什么？ |
| 方法论 | 该领域的主流方法论是否存在范式锁定？是否存在被排斥但可能更有效的方法论路径？ |
| 本体论 | 该事项的核心实体是什么？这些实体的存在方式（being）是否被恰当地理解？ |
| 价值论 | 事项描述中隐含了谁的价值立场？不同价值体系对该事项的判断会如何分歧？ |
| 系统论 | 事项作为更大系统的组成部分，其功能定位是否被正确理解？系统的涌现特性是什么？ |
| 历史唯物主义 | 该事项的历史演进逻辑是什么？经济基础与上层建筑的互动关系如何？ |
| 批判理论 | 该事项的权力结构是什么？谁的声音被放大，谁的声音被压制？解放的可能性在哪里？ |
| 第一性原理 | 若将事项还原到最基本的构成要素，其不可再分的底层逻辑是什么？ |
| 认知盲区 | 该领域中被默认接受但可能错误的常识是什么？跨学科视角能揭示哪些盲区？ |

**动态拓展原则**：
- 以上清单仅为哲学理论视角的示例，应根据事项的领域特征、理论传统、复杂度，动态识别适合的视角
- 简单事项可能只需 1-2 个视角，复杂事项可能需要 3-4 个
- 关键是选取「最能从宏观层面穿透该事项表象」的理论视角，而非纠结细节片段中的哲学问题
- 鼓励引入跨学科理论视角（如复杂性科学、演化论、现象学、解构主义等）

**执行要求**：哲学审视的洞察必须在 main.md 的「维度间逻辑关系」部分有所体现——说明某个维度的设计是如何受哲学审视结论驱动的。

---

#### 3.2.3 维度推导

基于属性分析，推导出适配的研究维度。

**维度推导原则**：
1. **逻辑推导原则**——维度必须从事项的属性分析中逻辑推导得出，而非从预设框架中选取
2. **互补覆盖**——各维度从不同角度切入事项，合在一起形成完整覆盖
3. **层次递进**——维度之间应存在逻辑递进关系（如宏观→中观→微观、原因→过程→结果）
4. **粒度适中**——维度数量通常 3~7 个

**子专题设计**：
- 每个维度下设置 2~6 个子专题，确保研究深度
- 子专题应互不重叠，合在一起覆盖该维度的全部关键面向
- 子专题的数量和内容由维度属性驱动

#### 3.2.4 扩展研究决策

基于事项属性判断是否需要扩展研究阶段。

**需要扩展研究的信号**：
- 多个维度之间存在显著交叉关联，单维度视角无法捕获
- 事项中存在需要跨维度综合才能回答的核心子问题
- 专题研究阶段可能产生维度间矛盾，需要横向审视
- 事项本身具有对比性质，需要跨维度的系统性对比框架

**扩展研究类型**（非穷举，由事项属性驱动）：横向对比、案例深潜、趋势推演、跨域关联、模式识别等。

**不需要扩展研究的情况**：维度间相对独立、综合判断阶段即可完成跨维度整合。

#### 3.2.5 专家体系设计

为每个维度和每项特殊职能配置专家智能体。

| 专家类别 | 数量 | 职责 | 命名格式 |
|---------|------|------|---------|
| 维度专家 | 每维度 1 位 | 该维度的独立深度研究 | `agent-{NN}-{专业领域}.md` |
| 横向研究专家 | 0~N 位（按需） | 扩展研究（如有） | `agent-{NN}-crosscut.md` |
| 综合判断专家 | 1 位 | 整合所有研究成果，形成整体判断 | `agent-{NN}-synthesis.md` |
| 质量审查专家 | 1 位 | 三级质量审查和事实核查 | `agent-{NN}-review.md` |
| 报告撰写专家 | 1 位 | 最终报告的整合与成稿 | `agent-{NN}-report.md` |

**专家定义要求**——每份定义文件必须具体且适配当前事项，内容结构见 `L1-engine-spec.md` §2.2 的专家定义文件规范。

### 3.3 引擎构造流程

完成事项解构后，按以下顺序生成引擎实例。

#### 步骤 1：创建目录结构

在 `tasks/` 下创建引擎实例目录。目录结构遵循 `L1-engine-spec.md` 的架构规范（通过 L2 `rules/engine-ref.md` 转化模板获取），包括：
- 维度研究区 `01` ~ `{D}`
- 扩展研究区（如有）`{D+1}` ~ `{D+E}`
- 综合判断区、质量控制区、最终报告区
- 每个研究区下创建 `data/sources-registry.md` 模板
- 创建 `progress.md` 初始文件

**命名规范**：遵循 `L1-common.md` §5。

#### 步骤 2：生成 main.md

`main.md` 是引擎的运行入口和主路由，其完整内容规范见 `L1-engine-spec.md` §2.2。构造时须确保：
- 包含事项描述、专家清单、维度总览、工作流路由表、执行纪律、全局配置六大模块
- 专家定义文件路径引用正确（相对路径，指向 `agents/` 目录）
- 规范引用指向 `./rules/`（L3 层内部路径，不得引用 L2 层路径）
- 维度间逻辑关系已明确阐述，且与哲学审视的洞察有对应关系

#### 步骤 3：生成专家定义文件

在 `agents/` 目录下为每位专家生成定义文件。每份定义必须具体适配当前事项，禁止泛泛通用描述。

#### 步骤 4：创建子文件骨架

为每个维度研究区创建空的子专题文件和综合评估文件骨架，写入：
- 文件用途说明
- 对应专家（引用 agent 文件路径）
- 预期产出格式提示

#### 步骤 5：动态生成任务专属 rules/

基于 L2 `rules/` 下的通用模板，为当前任务生成专属的 `{TASK}/rules/`，具体要求见本文件 §2.2（任务专属 rules/）。

### 3.4 输出规范与验收自检

完成引擎构造后，从结构合规和引擎质量两个层面逐项自检。

#### 3.4.1 结构合规

| 验收项 | 标准 |
|-------|------|
| 职责边界 | 引擎中不包含任何「执行研究」的逻辑——只有任务定义、专家定义和工作流路由 |
| 动态适配 | 维度、专家、子专题、流程均由当前事项属性驱动，无任何硬编码 |
| 可执行性 | 另一个 AI 读完 `main.md` 后能立即开始执行研究，无需额外解释 |
| 完整性 | 所有必要文件已生成，所有引用路径有效 |
| 专家质量 | 每位专家定义具体且适配当前事项，不存在泛泛通用描述 |
| L3 自包含 | `{TASK}/rules/` 已生成且内容完整，引擎 AI 无需访问 L2 层 |
| 工作流完整 | 从专题研究到最终成稿的完整路径已规划，阶段衔接条件已明确 |

#### 3.4.2 引擎质量

| 验收项 | 标准 |
|-------|------|
| 事项适配度 | 维度体系是从该事项的内在结构中涌现的，而非套用通用框架。换一个事项，维度体系应完全不同 |
| 研究深度 | 每个维度的子专题能引导出实质性的深度研究，而非停留在表面概述。子专题问题应具体、可调查、有明确的研究边界 |
| 研究广度 | 维度合在一起能形成对事项的完整覆盖，不存在明显的视角盲区或重要面向的缺失 |
| 哲学审视质量 | 属性分析中的哲学审视已实质性地影响了维度设计——而非仅作为装饰性文字存在。应能看到哲学审视的洞察如何转化为具体的维度或子专题设计决策 |
| 专家定义质量 | 每位专家的身份、思维立场、方法、数据源策略均与其负责的维度深度匹配，而非泛泛通用描述。不同专家之间应有明显的能力和认知模式差异化 |
| 维度间逻辑 | 维度之间存在清晰的逻辑递进关系（而非简单并列），并且这种关系已在 main.md 中明确阐述 |
| 扩展研究决策 | 扩展研究的「要/不要」决策有明确理由，且与事项的实际复杂度相称——不能对复杂事项省略，也不能对简单事项过度设计 |
| 防御性设计 | 引擎包含对常见退化模式的明确防御（禁止跨步、禁止占位内容、禁止泛泛通用描述），且这些约束具体可查、不是笼统口号 |
