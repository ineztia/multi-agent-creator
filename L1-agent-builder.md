# 构造多元专家研究引擎生成器的元指令

> 元声明
> - 版本：v1.0
> - 修订日期：2026-04-21
> - 实体层级：L1 元指令层
> - 读者：执行「构造生成器」任务的 AI（以下简称「你」）
> - 输入：本文件 + 同层的 `L1-agent-spec.md`、`L1-engine-spec.md`、`L1-research-spec.md`、`L1-common.md`
> - 输出：L2 生成器层全部文件——`AGENTS.md`（生成器入口）+ `rules/`（通用规范参考模板）
> - 边界：本文件只定义构造过程本身，不定义生成器方法论、引擎规范或研究成果规范

## 目标

构造一个**多元专家研究引擎生成器**——一个提示词系统，其唯一职责是接收用户输入的任意研究/分析/决策事项，分析事项属性，构造一个量身定制的多元专家研究引擎实例，输出可被另一个 AI 直接加载执行的引擎文件。该引擎将模拟组织多元专家体系，对用户指定的事项进行多维度深度研究，产出有价值的研究成果（如学术报告、决策参考报告）。

**核心约束**：生成器只负责构造引擎，不执行研究、不产出研究成果。

## 架构速览

```
L1 元指令层（本层）
├── L1-agent-builder.md    ← 你正在阅读的文件：构造过程指南
├── L1-agent-spec.md       ← 生成器能力/方法论/约束
├── L1-engine-spec.md      ← 引擎结构/流程/质量标准
├── L1-research-spec.md    ← 研究成果写作标准/范例
└── L1-common.md           ← 跨文件共享规范（内容标准、数据源评估）
         ↓
L2 生成器层（由你产出）
├── AGENTS.md              ← 生成器入口提示词
└── rules/                 ← 通用规范参考模板
         ↓
L3 引擎层（生成器运行时产出）
└── tasks/{TASK}/
         ↓
L4 成果层（引擎执行时产出）
└── 研究报告文件
```

| 层级 | 实体 | 由谁产出 | 由谁消费 |
|------|------|---------|----------|
| L1 元指令层 | 本文件 + 同层规范文件 | 人类编写 | 构造 AI 读取并执行 |
| L2 生成器层 | `AGENTS.md` + `rules/` | 构造 AI 产出 | 用户交互时由生成器 AI 加载执行 |
| L3 研究引擎层 | `{TASK}/main.md` + `{TASK}/rules/` 等 | 生成器运行时产出 | 另一个 AI 加载并执行研究 |
| L4 研究成果层 | 研究报告文件 | 引擎执行时产出 | 人类阅读使用 |

**层间隔离约束**：见 `L1-common.md` §1。每层自包含，不得引用其他层的文件。同层内文件可互相引用。

## 1. 构造什么

你的唯一任务是：读取 L1 层的全部规范，产出 L2 层的全部文件。所有后续工作均服务于「构造多元专家研究引擎生成器」这一目标。

### 1.1 L2 交付物清单

| 交付物 | 说明 |
|-------|------|
| `AGENTS.md` | 生成器入口提示词。AI 加载后即可接收用户事项、构造引擎实例 |
| `rules/*.md` | 通用规范参考模板。文件数量、命名和组织方式由你根据 L1 规范的理解自主设计——可以按职能拆分、按主题合并、或采用任何你认为最优的结构，只要能完整支撑生成器的工作即可 |

### 1.2 L2 层的核心约束

- **自包含**：L2 层的所有文件不得引用 L1 层文件。`AGENTS.md` 只引用 `./rules/` 内的文件
- **完整性**：`AGENTS.md` + `rules/` 合在一起，足以让生成器 AI 完成从事项到引擎的全部构造工作
- **可执行性**：AI 加载 `AGENTS.md` 后即可直接工作，无需额外解释或外部依赖

## 2. 构造方法论

### 2.1 准备——理解 L1 规范

按以下顺序阅读并理解 L1 层全部文件：

1. **`L1-common.md`**（共享规范）——理解跨文件共享的约束和标准
2. **`L1-agent-spec.md`**（生成器规范）——理解生成器应具备的能力、方法论和约束
3. **`L1-engine-spec.md`**（引擎规范）——理解引擎应具备的结构、流程和质量标准
4. **`L1-research-spec.md`**（研究成果规范）——理解研究成果的写作标准和范例

**理解检查点**：阅读完成后，在脑中确认以下问题的答案：
- 生成器的核心职责边界是什么？（不负责什么）
- 引擎的五阶段工作流是什么？
- 内容实质性标准的最低要求是什么？
- 数据源评估的权威性评级标准是什么？
- L2 层需要产出哪些文件？

若以上任一问题无法回答，返回重新阅读对应章节。

### 2.2 构造 AGENTS.md

将 `L1-agent-spec.md` 中定义的生成器规范转化为可执行的提示词系统：

**配置类**：
- **角色定义**：生成器的身份、职责边界
- **规范引用**：指向 `./rules/` 下的转化模板（不引用 L1 文件）

**流程类**：
- **输入处理**：事项接收、充分性检查、追问机制
- **事项解构**：属性分析、维度推导、专家体系设计
- **引擎构造流程**：
  - 目录创建
  - `main.md` 生成
  - 专家定义
  - 骨架文件生成
  - 动态生成 `{TASK}/rules/`
- **输出规范**：交付标准与验收自检

### 2.3 构造 rules/ 转化模板

将 L1 层规范文件的核心知识转化为 L2 层可直接使用的参考模板。

**转化输入**：L1 层全部规范文件（`L1-engine-spec.md`、`L1-research-spec.md`、`L1-common.md`）

**转化输出**：`rules/` 目录下的一组模板文件

**转化原则**：
- L1 文件是「规范定义」（是什么、为什么），L2 rules/ 是「操作模板」（怎么做、做成什么样）
- 转化后的模板必须自包含，不引用 L1 文件

**文件组织自主性**（受约束）：

rules/ 的文件数量、命名和内部组织由你自主设计，但须满足以下**最小约束**：

| 约束项 | 要求 |
|-------|------|
| 最小覆盖 | rules/ 必须包含 `engine-ref`、`research-ref`、`data-source-eval`、`quality-assurance` 四类规范 |
| 命名规范 | 遵循 `L1-common.md` §5（连字符分隔，不使用下划线） |
| 自包含 | 转化后的模板不得引用 L1 层文件路径 |
| 覆盖完整 | 模板合在一起须完整覆盖 L1 规范的全部核心知识 |

设计思路：不要模仿 L1 层的文件结构，而是从「生成器工作时需要查阅哪些参考」的角度出发，设计最合理的模板拆分方式。例如：
- 可以按职能拆分（引擎结构、写作规范、质量控制…）
- 可以按使用阶段拆分（构造期参考、执行期参考…）
- 可以合并相关小主题、拆分复杂大主题

L2 层 rules/ 只承载跨任务通用的规范模板，不涉及任何任务特定内容。任务专属的规范增删、领域适配由 L3 层 `{TASK}/rules/` 在引擎构造阶段动态完成。

### 2.4 确保 L3 生成能力

在 `AGENTS.md` 的引擎构造流程中，必须包含以下步骤：

- 生成器基于 `rules/` 下的通用模板，为每个具体任务动态生成 `{TASK}/rules/`
- 任务专属的 `rules/` 应适配具体事项（如研究领域、维度结构），而非照搬通用模板
- 生成后的 `{TASK}/rules/` 使 L3 层完全自包含，引擎 AI 无需访问 L2 层

### 2.5 冲突解决与异常处理

当构造过程中遇到以下情况时，按此协议处理：

| 场景 | 处理协议 |
|------|----------|
| L1 文件之间规定矛盾 | 以 `L1-agent-spec.md` > `L1-engine-spec.md` > `L1-research-spec.md` > `L1-common.md` 的优先级顺序为准 |
| L1 规范存在模糊/不可执行的部分 | 按「最接近规范意图的合理推断」执行，并在最终交付物中用 "> 注：" 标注该推断及原因 |
| 发现 L1 规范有明显遗漏 | 不自行发明新规则，在交付物中标注「L1 规范未覆盖此场景」并继续完成其余部分 |
| 用户对 L1 规范提出质疑 | 超出本文件边界，不处理 |

## 3. 验收标准

完成构造后，从结构合规和构造质量两个层面逐项自检。

### 3.1 结构合规

| 验收项 | 标准 |
|-------|------|
| 层间隔离 | L2 层无任何文件引用 L1 层路径 |
| 自包含性 | `AGENTS.md` + `rules/` 合在一起足以独立工作 |
| 职责边界 | `AGENTS.md` 只定义生成器行为，不包含执行研究的逻辑 |
| 转化完整性 | L1 规范的核心知识已完整转化为 `rules/` 模板，无关键规则缺失 |
| L3 生成能力 | `AGENTS.md` 中包含动态生成 `{TASK}/rules/` 的明确步骤 |

### 3.2 构造质量

| 验收项 | 标准 |
|-------|------|
| 指令精确性 | `AGENTS.md` 中的每条指令都具体、无歧义、可直接执行。不存在「适当」「合理」「必要时」等模糊描述 |
| 方法论保真 | `AGENTS.md` 的事项解构流程完整保留了 `L1-agent-spec` 的核心方法论（属性分析→维度推导→扩展研究决策→专家体系设计），无关键步骤缺失或简化 |
| 转化保真 | `rules/` 模板忠实传达了 L1 规范的核心约束与质量标准，未丢失关键规则、未降低标准、未引入与 L1 矛盾的新规则 |
| `rules/` 组织合理性 | `rules/` 的文件拆分服务于生成器的实际工作流程，而非简单镜像 L1 文件结构 |
| 情景适应性 | 用至少两个差异显著的假想事项（如一个决策型、一个研究型）心理走查 `AGENTS.md`，确认生成器能产出结构合理、维度差异化的引擎实例，而非套用固定模板 |
| 防御性设计 | `AGENTS.md` 包含对常见退化模式的明确防御——禁止跨步、禁止占位内容、禁止泛泛通用描述等约束已写入且具体可查 |
| 端到端可执行 | AI 加载 `AGENTS.md` 后，能从接收事项到交付完整引擎实例，全程无需人类提示或补充解释 |
