## 1. 核心设计原则

### 1.1 动态适配原则（统领全文）

引擎的一切设计决策均由事项属性驱动，不预设任何具体内容：

- 研究维度的数量、类型、层次——由事项属性决定
- 专家智能体的数量、类型、配置——由事项属性决定
- 子专题的数量和内容——由事项属性决定
- 扩展研究是否需要及其类型——由事项属性决定

> 本原则贯穿全文，后续各节不再逐一重复。

### 1.2 核心功能

在引擎中模拟组织多元专家体系，对用户指定的事项进行多维度深度研究，产出有价值的研究成果。

- **事项**：用户输入时确定，包括决策事项、科研课题、问题解答等任意需求
- **成果形式**：默认以符合学术期刊标准的学术报告呈现，用户另有指定时从其指定

## 2. 引擎架构

### 2.1 目录结构

**引擎实例目录**：引擎的工作目录为其根目录（即 `./`），内部所有路径均为相对路径。目录的创建位置和命名由生成器决定，引擎无需感知。

维度编号从 `01` 开始递增，扩展研究（如有）紧随其后，综合判断、质量控制、最终报告依次递增。设维度数为 D，扩展研究占 E 个编号（无扩展研究时 E=0），则：

| 区域 | 编号范围 |
|------|---------|
| 维度研究区 | `01` ~ `{D}` |
| 扩展研究区（可选） | `{D+1}` ~ `{D+E}` |
| 综合判断区 | `{D+E+1}` |
| 质量控制区 | `{D+E+2}` |
| 最终报告区 | `{D+E+3}` |

```
./
├── main.md                              # 引擎主入口
├── progress.md                          # 执行进度追踪
├── agents/                              # 专家智能体定义
│   ├── agent-01-{专家类型}.md
│   ├── agent-02-{专家类型}.md
│   └── ...
├── rules/                               # 任务专属规范（文件组织由生成器自主设计）
│   └── ...
├── 01-{维度1名称}/                      # 维度 1 研究区
│   ├── 01-01-{子专题1}.md
│   ├── 01-02-{子专题2}.md
│   ├── 01-{维度1名称}-综合评估.md
│   └── data/
│       └── sources-registry.md
├── 02-{维度2名称}/                      # 维度 2 研究区
│   └── ...
├── {D+1}-{扩展研究类型}/                # 扩展研究区（可选）
│   ├── {D+1}-01-{研究主题1}.md
│   └── data/
│       └── sources-registry.md
├── {D+E+1}-综合判断/                    # 综合判断区
│   ├── {D+E+1}-01-{综合判断内容1}.md
│   └── ...
├── {D+E+2}-质量控制/                    # 质量控制区
│   ├── review-round1.md
│   ├── review-round2.md
│   ├── review-round3-综合报告.md
│   └── fact-check-log.md
└── {D+E+3}-最终报告/                    # 最终报告区
    ├── {D+E+3}-{事项名称}-v1.0.md
    ├── {D+E+3}-摘要.md
    ├── {D+E+3}-附录-数据源清单与质量评级.md
    └── {D+E+3}-附录-关键学术参考文献.md
```

**命名规范**（通用规则见 `L1-common.md` §5）：
- 编号格式：目录编号两位数字 `XX`，子文件编号 `XX-YY`
- 综合评估文件命名：`XX-{维度名称}-综合评估.md`（不使用子编号）

### 2.2 核心文件规范

#### main.md（引擎主入口）

引擎的运行入口和主路由，应包含：

- **事项描述**：用户输入的原始事项和解构后的研究问题
- **专家智能体清单**：列出所有专家及其负责的维度，引用 `agents/` 下的定义文件
- **研究维度总览**：各维度名称、子专题列表、对应专家
- **工作流路由表**：各阶段的执行顺序、衔接条件、文件产出路径
- **全局配置**：成果形式、质量标准等级、扩展研究决策结果、规范引用指向 `./rules/`
- **执行纪律与进度追踪**：引用 `./rules/` 中的引擎执行纪律参考，说明逐文件串行执行规则、内容实质性标准、阶段门控验证方法和恢复协议

#### agents/agent-XX-{专家类型}.md（专家智能体定义）

每份专家定义文件应包含：

- **专家身份**：专业领域、资历背景、核心能力
- **思维立场**：该专家看问题的独特认知视角和追问方式——不是「使用什么方法」，而是「会问什么别人不会问的问题」「会从什么角度质疑常识」「面对数据不足时如何处理」。不同专家之间的差异化不仅体现在研究主题上，更应体现在认知模式上
- **研究范围**：负责的维度和子专题
- **研究方法**：该领域的主要研究方法和数据来源策略
- **产出要求**：应产出的文件清单和格式规范
- **质量标准**：该领域特有的质量评判标准

#### progress.md（执行进度追踪）

引擎运行期间自动维护的状态文件，用于记录工作流执行进度，支持任务中断后恢复。

**文件格式**：

```markdown
# 执行进度

## 状态概要

- 当前阶段：[阶段一：专题研究 / 阶段二：扩展研究 / ... / 阶段五：最终成稿]
- 当前专家：[agent-XX-{类型}]
- 当前文件：[正在处理的文件路径]
- 最后更新：[时间戳]

## 文件完成清单

| 阶段 | 文件 | 状态 | 完成时间 |
|------|------|------|----------|
| 阶段一 | 01-01-{子专题}.md | ✅ 已完成 / ⏳ 进行中 / ⬜ 未开始 | [时间] |
| ... | ... | ... | ... |

## 阶段门控记录

| 门控点 | 状态 | 检查时间 | 备注 |
|-------|------|---------|------|
| 阶段一 → 阶段二 | ⬜ 未到达 | - | - |
| ... | ... | ... | ... |
```

**更新规则**：

| 触发时机 | 更新内容 |
|---------|----------|
| 开始处理一份新文件前 | 将该文件状态改为 `⏳ 进行中`，更新「当前专家」和「当前文件」 |
| 一份文件写入完成后 | 将该文件状态改为 `✅ 已完成`，记录完成时间 |
| 通过阶段门控验证后 | 更新「当前阶段」，记录门控通过时间 |
| 质量审查打回修改时 | 在门控记录中标注打回信息和责任文件 |

**恢复协议**：

执行 AI 在启动时（包括中断后重新启动），必须执行以下恢复流程：

1. **检查 `progress.md` 是否存在**：若存在，说明是中断恢复场景
2. **读取状态概要**：确认当前阶段和当前文件
3. **扫描文件完成清单**：找到第一个状态为 `⏳ 进行中` 或 `⬜ 未开始` 的文件
4. **从该文件继续执行**：如果文件状态为 `⏳ 进行中`，检查文件内容——若已有实质内容则标记完成并继续下一个，若仍为骨架则重新研究该文件
5. **不得重复已完成的工作**：状态为 `✅ 已完成` 的文件直接跳过

#### data/sources-registry.md（数据源注册表）

每个维度研究区下的数据源注册表应包含：

| 字段 | 说明 |
|------|------|
| 来源名称 | 数据源的完整名称 |
| 类型 | 学术论文/行业报告/政府数据/新闻报道等 |
| 权威性评级 | 高/中/低 |
| 访问时间 | 数据获取时间 |
| 对应文件 | 在 `data/` 目录中的文件名 |

## 3. 研究维度与专家体系

### 3.1 维度设计要求

- 每个维度配置专门的专家智能体
- 每个维度下设置多个子专题，确保研究深度
- 维度的数量、类型、层次完全由事项属性驱动

### 3.2 维度研究独立性

- **专题研究阶段**：各维度专家独立研究，互不干扰。每位专家独立采集数据、独立论证、独立产出研究报告，均指向核心问题，不做维度间横向交叉
- **扩展研究阶段**（例外）：扩展研究允许且鼓励跨维度综合

## 4. 产出层次结构

引擎采用三层递进的产出结构：

| 层级 | 产出类型 | 说明 | 文件命名 |
|------|---------|------|----------|
| 第一层 | 子专题研究报告 | 可独立阅读，包含完整研究过程和结论 | `XX-YY-{专题名称}.md` |
| 第二层 | 维度综合评估 | 整合该维度下所有子专题的研究成果 | `XX-{维度名称}-综合评估.md` |
| 第二层 | 扩展研究报告 | 跨维度综合分析（如有） | `{D+1}-01-{研究主题}.md` 等 |
| 第三层 | 综合判断报告 | 整合所有研究成果的整体判断 | `{D+E+1}-01-{内容}.md` 等 |
| 第三层 | 最终报告 | 完整的研究成果交付物，含报告、摘要、附录 | `{D+E+3}-{事项名称}-v1.0.md` 等 |

## 5. 研究工作流

### 5.1 阶段总览

```
专题研究 ──→ [扩展研究（可选）] ──→ 综合判断 ──→ 质量审查 ──→ 最终成稿
             ↑ 由事项属性决定是否需要
```

### 5.2 专题研究阶段（必选）

**目标**：从多个维度对事项进行深度并行研究

**要求**：
- 各维度专家独立、并行研究，互不干扰
- 每个维度下设置多个子专题
- 每份独立研究需查询尽可能多的权威信息，形成研究依据

**产出**：各维度的子专题研究报告 + 维度综合评估

### 5.3 扩展研究阶段（可选）

**触发条件**：由事项属性决定。

**研究类型**（非穷举）：横向对比、案例深潜、趋势推演、跨域关联等。

**要求**：
- 允许跨维度综合信息
- 应发现专题研究阶段未覆盖的洞察

**产出**：扩展研究报告

### 5.4 综合判断阶段（必选）

**目标**：整合所有研究成果，形成整体判断

**要求**：
- 整合专题研究和扩展研究（如有）的成果
- 回答核心问题，形成整体判断
- 研判未来趋势，提出建议

**产出**：综合判断报告

### 5.5 质量审查阶段（必选）

质量审查是工作流的关键环节，详细规范在本文件 §6。

### 5.6 最终成稿阶段（必选）

**目标**：构建整个研究的顶层论证——读者仅阅读最终报告，即可理解从证据到核心结论的完整推理过程

**核心要求——论证重建而非结论搬运**：

最终报告不是将子专题的结论搬运到一个文件中，而是将分散在子专题中的推理链**重建**为一个连贯的顶层论证。两者的关键区别：

> **结论搬运**（不合格）：「维度01发现社消零按卖方所在地统计；维度02发现制造业企业有隐性零售收入；维度03发现工业品电商不计入社消零；因此建议……」——读者知道结论，但不知道为什么。
>
> **论证重建**（合格）：在最终报告中重新展开关键推理链——社消零按卖方所在地统计（§X），这意味着龙田的社消零「薄弱」不是没有零售活动，而是零售活动未被捕获（推理步骤）；制造业企业中存在隐性零售收入（§Y），通过产销分离可将这些收入显性化（推理步骤）；但产销分离的贸易公司是否留在龙田取决于「留驻理由」（约束条件）；因此核心策略不是泛化招商，而是针对性构建留驻激励（修正结论）——读者即使不看子专题，也能理解策略的逻辑根基。

**具体要求**：

1. **关键推理链必须在最终报告中完整呈现**——每个核心策略建议必须展示从制度事实→机制推理→约束条件→修正结论的完整路径，不能只写结论再标注「详见XX-YY」
2. **约束条件和边界条件不可省略**——子专题中识别出的约束（如产销分离必须基于真实商业目的、500万门槛的统计隐形效应）必须在最终报告中保留，因为它们决定了策略的适用边界
3. **争议和不确定性不可省略**——子专题中讨论的替代解释、估算的不确定性、数据的局限，必须在最终报告中保留，因为它们决定了结论的可信度
4. **摘要和附录是补充，不是替代**——摘要是最终报告的压缩版（不是结论清单），附录提供数据细节

**产出**：最终研究报告、摘要、附录

### 5.7 阶段衔接规则

| 前序阶段 | 衔接条件 | 后续阶段 |
|---------|---------|---------|
| 专题研究 | 所有维度研究完成 → 判断是否需要扩展研究 | 扩展研究 或 综合判断 |
| 扩展研究 | 扩展研究完成 | 综合判断 |
| 综合判断 | 综合判断完成 | 质量审查 |
| 质量审查 | 审查通过（或达到最大审查轮次） | 最终成稿 |

### 5.8 执行纪律（防跳步规范）

> **为什么需要本节**：部分 AI 在执行引擎时，会创建目录和文件骨架后，跳过实际研究过程，直接生成最终报告。本节定义强制执行纪律，确保每个研究步骤被真正执行。

#### 5.8.1 逐文件串行执行

- **一次只处理一个文件**：每次以一位专家的身份，完成一份子专题研究报告的完整写作，写入文件后，才能开始下一份
- **禁止批量生成**：不得在脑中「一次性构思所有维度的研究」然后批量输出
- **禁止占位内容**：最终文件不得保留任何 `[待研究]`、`[TODO]`、`[此处补充]` 等占位标记

#### 5.8.2 内容实质性标准

内容实质性标准定义见 `L1-common.md` §2。该标准须通过 L3 `{TASK}/rules/` 转化为引擎可直接参考的检查项。

引擎执行 AI 应在生成每份文件前，对照 `./rules/` 中的内容实质性检查项逐条确认，确认通过后方可写入文件。任何不满足最低要求的文件均视为未完成。

#### 5.8.3 阶段门控验证

进入下一阶段前，执行 AI 必须逐一检查前序阶段的每份产出文件：

| 门控点 | 验证内容 | 不通过处理 |
|-------|---------|-----------|
| 专题研究 → 下一阶段 | 每份子专题报告和综合评估是否满足 §5.8.2 标准 | 回到未完成的文件继续研究 |
| 扩展研究 → 综合判断 | 扩展研究报告是否包含对专题研究成果的实际引用 | 回到扩展研究补充 |
| 综合判断 → 质量审查 | 综合判断报告是否实际整合了前序阶段的具体发现 | 回到综合判断补充 |

#### 5.8.4 强制执行顺序（含进度追踪）

```
0. 启动时检查 progress.md → 若存在则执行恢复协议（§2.2）→ 从中断点继续

对每个维度 D_i（i = 1..D）：
  1. 读取该维度专家定义文件，进入专家角色
  对 D_i 下的每个子专题 S_j（j = 1..N）：
    2. 更新 progress.md：当前文件 → ⏳ 进行中
    3. 针对子专题 S_j 进行独立研究
    4. 将完整研究报告写入 XX-YY-{子专题}.md
    5. 更新 data/sources-registry.md
    6. 更新 progress.md：当前文件 → ✅ 已完成
  完成所有子专题后：
    7. 更新 progress.md：综合评估文件 → ⏳ 进行中
    8. 基于已完成的子专题报告，撰写维度综合评估
    9. 将综合评估写入 XX-{维度名称}-综合评估.md
   10. 更新 progress.md：综合评估文件 → ✅ 已完成

所有维度完成 → 门控验证（§5.8.3）→ 更新 progress.md → 进入下一阶段
```

**严禁以下行为**：
- 创建完目录结构后直接跳到综合判断或最终报告
- 用概括性语言代替实际研究
- 多个维度共享相同的分析框架和结论
- 在未完成前序阶段全部产出的情况下启动后续阶段
- **禁止无推理断言**：任何结论性判断必须展示推理过程，不得直接从数据跳到结论
- **禁止信息陈列替代论证**：每个实质性章节的核心内容必须以论述文本而非纯表格/列表呈现
- **禁止单一解释**：对于重要的因果判断，必须考虑替代解释并说明取舍理由
- **禁止虚假精确**：推测性数据不得以确定性格式（如权威性评级表格）呈现

#### 5.8.5 研究迭代机制

当前「逐文件串行、写完即完成」的执行纪律确保了研究过程的有序性，但不应成为思维深度的桎梏。以下场景允许且鼓励回溯修正：

- **维度综合评估阶段**：撰写综合评估时，若发现子专题报告中存在不准确、论证链断裂或结论需要修正之处，应回溯修正子专题报告
- **综合判断阶段**：若发现前序研究存在明显薄弱环节（如某个关键因果链缺乏论证），应触发补充研究
- **质量审查阶段**：审查发现「论证链断裂」「估算值无依据」时，应打回要求补充推理，而非仅打回格式

回溯修正时，在 `progress.md` 中记录修正原因和涉及的文件。

## 6. 质量控制规范

### 6.1 质量属性

引擎产出的研究成果应满足以下质量属性：

- **学术规范性**：研究报告严格符合学术规范，是完整文章而非提纲
- **逻辑完整性**：研讨过程层次清晰、论证严密
- **数据可靠性**：数据来源权威，事实准确
- **创新性**：有独到见解和创新点

### 6.2 三级审查制度

| 轮次 | 审查重点 | 产出文件 |
|------|---------|--------|
| 第一轮 | 结构完整性：各阶段产出是否齐全、格式是否合规 | `review-round1.md` |
| 第二轮 | 认知质量：推理是否成立、论证链是否完整、是否考虑替代解释 | `review-round2.md` |
| 第三轮 | 综合评审：整体一致性、创新性、实用价值 | `review-round3-综合报告.md` |

**第二轮审查——认知质量**

**审查方法的根本要求**：审查者必须先**理解**被审查报告的论证，然后才能**评价**该论证的质量。因此，第二轮审查的第一步不是打勾，而是复述——审查者必须证明自己读懂了。

**审查流程**：

**步骤1：论证复述**（理解验证）——对每份子专题报告，审查者须用自己的话复述其核心论证链：

> 报告[XX-YY]的核心论证是：[从A事实出发→推理得到B→但考虑C约束→修正为D结论]。该论证的关键前提是[……]，最薄弱的环节是[……]。

如果审查者无法完成复述，说明该报告的论证不清晰（这是报告的问题，不是审查者的问题）。复述失败 = 审查不通过。

**步骤2：论证评价**（基于理解的判断）——在复述基础上，评价以下维度：

- 推理链是否有跳步：复述中是否存在「从A直接到D」而缺少B、C？
- 替代解释是否被考虑：复述中是否有「选择了解释X」而没有提到为什么不是Y？
- 约束条件是否被识别：复述中是否包含结论的适用边界？
- 推测与实证是否混淆：复述中是否有「确定性地陈述了应标注为推测的内容」？

**步骤3：审查报告的写法**——第二轮审查报告不是打勾表，而是论证评估文档：

> 对报告[XX-YY]：
> - 核心论证复述：[……]
> - 论证最薄弱环节：[……]
> - 建议补充：[……]
>
> 对报告[XX-ZZ]：
> - 核心论证复述：[……]
> ……

**核心逻辑**：审查者无法复述的论证，就是不清楚的论证。审查的价值不在于打勾，而在于发现「哪里不清晰」。

### 6.3 事实核查

- 核查数据来源的权威性和时效性
- 核查关键事实的准确性
- 记录核查过程和结果到 `fact-check-log.md`

### 6.4 审查不合格处理

- 明确指出问题所在及对应的责任专家
- 打回对应专家重新修改
- 修改后重新提交审查
- **终止条件**：单份文件最多打回 2 次（共 3 次提交机会）
